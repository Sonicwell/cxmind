# IE 测试覆盖差距分析

> **问题**: 87 个已识别问题为什么全部未被现有测试发现？  
> **结论**: 主要是**测试类型缺失**（非脚本编写错误），现有测试只覆盖了正确路径（happy path）的功能验证

---

## 测试现状

| 指标 | 值 |
|------|-----|
| 测试文件 | 31 个 |
| 覆盖包 | 9/10（`pcap` 包 **零测试**） |
| 总测试函数 | ~50 个 |
| 通过率 | 100%（含 `-race`） |
| 测试类型 | 纯单元测试 |

---

## 根因分析：5 类缺失的测试类型

### 1. ❌ 并发竞争测试（影响 28 个问题）

**受影响问题**: P0-1, P0-2, P0-5, T1, T2, T7, T9, H1-H4, S1-S3, S6, PC1 等

现有唯一的并发测试 `TestDeadlockInjectRTPAndStop` 验证了 `sync.Map` 不会死锁——但这**不是实际的 bug**。真正的问题是：

```go
// deadlock_test.go — 测试了什么
stream.mu.Lock()           // ← 只测 sync.Map + stream.mu 不冲突
time.Sleep(10 * µs)
stream.mu.Unlock()

// 未测试的实际 bug
stream.Close()             // ← 包含 time.Sleep(2s) 阻塞！
// 在 monitorTimeouts 的 Range 回调中调用 → 阻塞整个超时检测
```

**缺失**: 无 `go test -race` 场景验证真实并发路径（BYE+timeout 竞争、连接池争用）

### 2. ❌ 集成/端到端测试（影响 25 个问题）

**受影响问题**: C1-C8, R1-R5, T3-T6, S9 等

| 组件 | 现有测试 | 实际覆盖 |
|------|---------|---------|
| ClickHouse | `sql_test.go` (2个) | **只验证 SQL 字符串语法**，不测试实际写入 |
| Redis | `callstate_test.go` + `policy_cache_test.go` | Mock 测试，不测真实 pipeline |
| HEP | `hep_test.go` (1个) | 只测 HEP3 解码，不测 TCP 服务器 |
| PCAP | **零测试** | 完全未覆盖 |
| RTP → CH | 无 | 不测 RTCP → ClickHouse 写入流程 |

ClickHouse 的 22,000 次/秒单条 INSERT 问题（C1-C5）不可能被纯字符串测试发现。

### 3. ❌ 负面/边界测试（影响 18 个问题）

**受影响问题**: P0-4, SP1-SP4, P1-10, P2-15, P2-22 等

| 测试文件 | 测试了什么 | 缺少什么 |
|---------|----------|---------|
| `sip_test.go` | 标准 INVITE + 200 OK 解析 | 多 Via header、折行、畸形消息 |
| `parser_edge_cases_test.go` | 格式变体 (extra spaces) | **验证了 bug 是"预期行为"** — garbage input 被当作 Request |
| `timeout_test.go` | 超时触发/不触发的 flag | 不测 `CloseAll` 2s 阻塞、heap 竞争 |
| `asr_handler_test.go` | final/non-final result 路由 | 不测 ClickHouse 阻塞导致的 goroutine 泄漏 |

特别值得注意的是 `parser_edge_cases_test.go`：

```go
// 这个测试 *确认* 了 SP3 的 bug 是预期行为
{
    name: "Garbage start",
    wantRequest: true,           // ← "预期"垃圾被当作 Request
    wantMethod:  "junk",         // ← 不触发错误
}
```

### 4. ❌ 性能/负载测试（影响 20 个问题）

**受影响问题**: S1-S15, C1-C5, W1-W3, PC3 等

没有任何 benchmark 测试（`func Benchmark*`）。无法发现：
- `MaxRecorders=1000` 在高并发下的限制
- Redis pool_size=50 的饱和
- ClickHouse 连接池耗尽
- GeoIP 无缓存的 CPU 开销

### 5. ❌ 资源泄漏测试（影响 12 个问题）

**受影响问题**: P0-2, T6, T8, M1, S10 等

没有测试验证：
- goroutine 泄漏（stopCh 覆盖）
- `terminated` map 无限增长
- `agentMonitoring` sync.Map 无 TTL 累积
- 文件描述符泄漏

## 结论

### 不是"测试脚本编写问题"——是测试策略缺失

现有 31 个测试文件编写质量尚可（使用 testify、mock、table-driven 等），但整体测试策略存在根本缺陷：

| 维度 | 现状 | 应有 |
|------|------|------|
| **测试金字塔** | 只有单元测试底层 | 需要集成测试 + 端到端测试 |
| **测试视角** | 只验证"做对了什么" | 需要验证"不做错什么" |
| **并发测试** | 1 个伪死锁测试 | 需要 `-race` + 真实竞争场景 |
| **性能测试** | 0 个 benchmark | 需要 `Benchmark*` + pprof |
| **覆盖率** | 估计 < 15% | 建议 ≥ 60% |

### 87 个问题的分类

| 测试缺失类型 | 受影响问题数 | 占比 |
|------------|-----------|------|
| 缺少并发竞争测试 | 28 | 32% |
| 缺少集成测试 | 25 | 29% |
| 缺少性能/负载测试 | 20 | 23% |
| 缺少边界/负面测试 | 18 | 21% |
| 缺少资源泄漏测试 | 12 | 14% |

> [!NOTE]
> 部分问题同时受多种测试缺失影响（如 T7 同时需要并发测试和集成测试），因此百分比总和 > 100%
